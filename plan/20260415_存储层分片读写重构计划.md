# 20260415 存储层分片读写重构计划

## 1. 说明

本计划用于重构当前应用的 `app_data` 服务层。

本次重构的核心约束已经明确：

1. 不改当前磁盘文件结构。
2. 不把结构化真源切换到 SQLite。
3. 不推翻现有 `config/state/chat/conversations` 布局。
4. 只重构 `app_data` 服务层的读写边界、缓存边界与后台持久化边界。

当前真正的问题不是“文件没拆”，而是：

1. 业务热路径仍然围绕整份 `AppData` 聚合对象做读改写。
2. 只改一个小功能，也会走整份 `AppData` 的写回或排队写回。
3. `agents`、`runtime`、`chat/index`、单个 `conversation` 明明已经是分文件布局，但服务层仍按整坨状态使用。

因此，本计划的目标不是“改布局”，而是：

1. 保留现有文件结构。
2. 停止业务热路径对整份 `AppData` 的依赖。
3. 把存储服务改成按分片读写、按分片缓存、按分片持久化。

本计划已获得用户确认，可进入实现阶段。

## 2. 当前问题定位

### 2.1 当前布局已经是分文件，但服务仍是整对象

当前布局已有以下分片：

1. `config/agents.json`
2. `state/runtime_state.json`
3. `chat/index.json`
4. `chat/conversations/<id>.json`

但实际业务入口仍然以以下函数为核心：

1. `read_app_data(path) -> AppData`
2. `write_app_data(path, data)`
3. `state_read_app_data_cached(state)`
4. `state_write_app_data_cached(state, data)`
5. `state_schedule_app_data_persist(state, data)`

这导致：

1. 写人格时会携带整份 `AppData`。
2. 改运行态字段时会携带整份 `AppData`。
3. 追加一条消息时也会把整份 `AppData` 推进后台持久化队列。

### 2.2 当前最浪费的地方在于“无关分片被跟着重写”

举例：

1. 只改 `assistant_department_agent_id`，理论上只该写 `runtime_state.json`。
2. 只保存人格，理论上只该写 `agents.json`。
3. 只给某个会话追加消息，理论上只该写该会话文件，必要时补写 `chat/index.json`。

但当前实际情况是：

1. 一次小改动会经过整份 `AppData`。
2. 最终 `write_app_data()` 会重写多个无关文件。
3. 后台持久化队列也只认识整份快照，不认识分片。

### 2.3 当前症结不在磁盘布局，而在服务边界

结论：

1. 现有文件结构本身已经足够支持第一阶段优化。
2. 当前最该重构的是 `app_data` 服务边界。
3. 只要把读写、缓存、持久化从“整份 `AppData`”改成“按分片”，性能和复杂度都会显著改善。

## 3. 目标

### 3.1 功能目标

1. 保存人格时只写 `agents.json`。
2. 修改运行态字段时只写 `runtime_state.json`。
3. 修改会话标题、状态、摘要时只写目标会话文件与必要的 `chat/index.json`。
4. 追加消息时只写目标会话文件与必要的 `chat/index.json`。
5. 删除会话时只删目标会话文件并更新 `chat/index.json`。

### 3.2 架构目标

1. `AppData` 不再作为业务热路径的唯一读写边界。
2. 新增分片级 store / repository 接口。
3. 缓存按分片组织，不再维护“整份应用数据缓存”作为主路径。
4. 后台持久化队列按分片 key 合并，而不是按整份快照合并。

### 3.3 性能目标

1. 热路径尽量减少无关 JSON 的序列化。
2. 会话消息追加不再牵连人格、运行态、其他会话。
3. 配置/人格保存不再牵连聊天数据。
4. 多个会话并发小改动时，后台持久化可以分片合并，而不是互相覆盖整份状态。

## 4. 非目标

1. 本次不改现有磁盘目录结构。
2. 本次不把会话消息拆成 chunk 或 jsonl。
3. 本次不把结构化真源迁到 SQLite。
4. 本次不重做前端交互。
5. 本次不重写 memory / task / delegate 的存储层。

说明：

1. 本次是服务边界重构，不是存储介质重构。
2. 若后续还要继续把单会话再拆成 `meta + messages`，应另立计划，不混入本次主线。

## 5. 最终方案

## 5.1 核心方案

保留现有文件结构，重构 `app_data` 服务为“分片读写模型”。

具体来说：

1. 保留 `agents.json`
2. 保留 `runtime_state.json`
3. 保留 `chat/index.json`
4. 保留 `chat/conversations/<id>.json`

但业务不再以整份 `AppData` 进行保存，而是改用：

1. `agents` 分片
2. `runtime_state` 分片
3. `chat_index` 分片
4. `conversation:<id>` 分片

## 5.2 AppData 的新定位

`AppData` 仍然保留，但定位改变为：

1. 兼容聚合视图
2. 启动快照组装结果
3. 导入导出与迁移辅助结构
4. 调试或少数全量逻辑使用

明确不再作为：

1. 聊天热路径保存模型
2. 人格保存模型
3. Runtime 保存模型
4. 后台持久化队列的基本单元

## 5.3 服务边界必须改成按分片

新服务层应提供以下方向的 API：

1. `read_agents()`
2. `write_agents(agents)`
3. `read_runtime_state()`
4. `patch_runtime_state(patch)`
5. `read_chat_index()`
6. `write_chat_index(index)`
7. `read_conversation(id)`
8. `write_conversation(id, conversation)`
9. `patch_conversation_meta(id, patch)`
10. `append_conversation_messages(id, messages)`
11. `replace_conversation_todos(id, todos)`
12. `replace_conversation_memory_recall(id, table)`
13. `delete_conversation(id)`

## 6. 分片模型设计

## 6.1 agents 分片

对应文件：

1. `config/agents.json`

职责：

1. 人格列表读写
2. 用户人格 / 系统人格保底逻辑
3. 私有工作区合并前的基础人格真源

约束：

1. 保存人格时只写这个文件。
2. 不应再通过整份 `AppData` 间接写入。

## 6.2 runtime_state 分片

对应文件：

1. `state/runtime_state.json`

职责：

1. `assistant_department_agent_id`
2. `user_alias`
3. `response_style_id`
4. `pdf_read_mode`
5. `background_voice_screenshot_keywords`
6. `background_voice_screenshot_mode`
7. `instruction_presets`
8. `main_conversation_id`
9. `image_text_cache`
10. `remote_im_contacts`

说明：

1. 本阶段继续保留这些字段在同一分片中。
2. 不额外改变文件结构。

## 6.3 chat_index 分片

对应文件：

1. `chat/index.json`

职责：

1. 维护会话列表所需摘要字段
2. 作为列表页的快速读取入口

字段包括但不限于：

1. `id`
2. `updated_at`
3. `status`
4. `summary`
5. `archived_at`

约束：

1. 只在会话头变更时更新。
2. 不应因人格保存、runtime 修改而重写。

## 6.4 conversation 分片

对应文件：

1. `chat/conversations/<id>.json`

职责：

1. 单会话完整数据读写

说明：

1. 在“不改文件结构”的前提下，单会话文件内部仍然是完整 `Conversation`。
2. 这意味着追加消息仍会重写当前会话文件。
3. 但这是可接受的，因为已经不再牵连其他会话和其他分片。

## 7. 缓存模型重构

## 7.1 需要淘汰的旧缓存思路

当前主缓存思路是：

1. `cached_app_data`
2. `cached_app_data_mtime`
3. `cached_app_data_dirty`

这个思路的问题是：

1. 它把所有分片绑成一个失效单元。
2. 一个小改动也会把“整份应用数据缓存”标记脏。
3. 无法针对单个会话做细粒度缓存更新。

## 7.2 新缓存边界

建议改为以下缓存：

1. `cached_agents`
2. `cached_agents_mtime`
3. `cached_runtime_state`
4. `cached_runtime_state_mtime`
5. `cached_chat_index`
6. `cached_chat_index_mtime`
7. `cached_conversations: HashMap<String, Conversation>`
8. `cached_conversation_mtimes: HashMap<String, SystemTime>`

## 7.3 缓存更新原则

原则如下：

1. 写某个分片后，只更新该分片缓存。
2. 不因单会话改动清空所有会话缓存。
3. 读取某个会话时，只比较该会话文件的 mtime。
4. 读取列表页时，只比较 `chat/index.json` 的 mtime。

## 8. 后台持久化模型重构

## 8.1 需要淘汰的旧持久化单元

当前后台持久化单元是：

1. `PendingAppDataPersist { seq, data: AppData }`

这会导致：

1. 队列里装的是整份快照。
2. 后来的小改动会覆盖前面的无关改动视图。
3. 合并只能按整份 `AppData` 合并。

## 8.2 新的持久化单元

应改为按分片 key 持久化。

建议 key 形式：

1. `agents`
2. `runtime`
3. `chat_index`
4. `conversation:<id>`

建议待写入结构：

1. `PendingShardPersist`
2. 内含 `shard_key`
3. 内含该分片最新值
4. 内含序号或时间戳用于去旧留新

## 8.3 持久化合并规则

规则如下：

1. 同一 `conversation:<id>` 在短时间内多次修改，只保留最新版本写盘。
2. `agents` 与 `runtime` 各自独立合并，不互相影响。
3. `chat_index` 可独立合并。
4. 不同会话的写入可以在同一轮 worker 中分别落盘。

## 8.4 写入顺序约束

为了减少列表与详情短暂不一致，建议：

1. 若某次变更同时改会话和索引，先写会话，再写 `chat_index`。
2. 删除会话时，先删会话文件，再更新 `chat_index`。

一致性保证与失败处理：

1. `conversation -> chat_index` 的更新必须设计为幂等。
2. 若会话文件写入成功但 `chat_index` 更新失败，禁止回滚已成功的会话文件，改为立即重试索引写入。
3. 即时重试必须保持在当前调用链内完成，不新增新的秒级或毫秒级定时调度；建议只做 1-2 次短重试，避免把当前 1ms 级写入路径重新拉回到旧的延迟模型。
4. 若短重试仍失败，则将该会话标记为 `index-pending`（内存态或运行态恢复队列皆可），等待后续显式修复流程处理。
5. 后台索引修复任务必须幂等，允许重复触发，不要求事务式回滚会话正文。
6. 若删除会话文件成功但索引更新失败，同样进入 `index-pending` 修复流程；不得尝试“恢复已删会话文件”作为默认补偿策略。
7. 运维处置以“晚些修正索引”为默认补偿方式，而不是“回滚已成功正文写入”。

触发与调用规则：

1. 当 `chat_index` 写入失败时，当前调用链只负责短重试与登记 `index-pending`，不额外引入新的高频定时器。
2. 应用启动时检查是否存在 `index-pending` 项，有则优先触发一次修复。
3. 与会话相关的后续写入路径在检测到 `index-pending` 时，可顺带触发一次轻量修复，但不得因此把热路径降回旧的延迟模型。
4. 管理员或开发者可以通过手动命令显式触发 `indexRebuild`。
5. `indexRebuild` 必须支持重复执行、支持锁保护，并在默认模式下避免长时间独占业务写入。

## 9. 兼容层设计

## 9.1 read_app_data 继续保留，但降级为聚合器

`read_app_data()` 可继续保留，但职责改变为：

1. 分别读取 `agents`
2. 读取 `runtime_state`
3. 读取 `chat_index`
4. 读取所有 `conversation`
5. 聚合出 `AppData`

说明：

1. 它不再是主热路径 API。
2. 它只作为兼容层或少数全量逻辑的聚合器存在。

## 9.2 write_app_data 继续保留，但降级为兼容写入器

`write_app_data()` 可在迁移期保留，但必须标记为：

1. 兼容层
2. 非热路径
3. 禁止新增调用点

代码级约束：

1. 函数符号添加 `#[deprecated(note = "...")]`。
2. 函数文档必须明确写明“仅用于兼容/迁移/导入导出，不得作为新热路径入口”。
3. 推荐替代入口必须写明为：
   `state_write_agents_cached` / `state_write_runtime_state_cached` / `state_write_chat_index_cached` / `state_write_conversation_with_chat_index_cached`
4. 迁移时间线明确为：
   新生产调用点立即禁止；兼容/测试/迁移用途暂时保留；待兼容路径完全收拢后再评估是否最终删除。

CI 约束：

1. 增加静态检查，阻止新增 `write_app_data()` 调用点。
2. 允许白名单仅限：
   函数定义本身、迁移回灌调用点、兼容层集中封装点、测试用途。
3. 若出现新的生产调用点，CI 直接失败。

说明：

1. 新逻辑不得再基于它开发。
2. 未来待所有调用迁出后，再决定是否删除。

## 10. 关键链路改造方案

## 10.1 配置与人格链路

目标：

1. 保存人格只写 `agents` 分片。
2. 保存会话 API 相关轻量运行态只写 `runtime_state` 或 `app_config`。

涉及方向：

1. `save_agents`
2. 与 `assistant_department_agent_id` 同步的 runtime 更新
3. 各类 chat settings 保存

原则：

1. 不再先读整份 `AppData` 再回写。

## 10.2 聊天发送链路

目标：

1. 发送用户消息时只更新目标会话。
2. 生成 assistant 消息时只更新目标会话。
3. 如需刷新列表摘要，再补写 `chat_index`。

原则：

1. 不再构造“更新后的整份 `AppData`”再丢给后台持久化。

## 10.3 归档链路

目标：

1. 归档本质是修改目标会话的 `summary/status/archived_at/...`
2. 完成后同步刷新 `chat_index`

原则：

1. 只改目标会话与索引，不碰其他会话。

## 10.4 会话删除链路

目标：

1. 删除目标会话文件
2. 更新 `chat_index`
3. 必要时修正 `main_conversation_id`

说明：

1. 这类链路虽然会同时改多个分片，但仍应是明确的“有限分片更新”，不是整份应用数据重写。

## 10.5 Remote IM 与图片缓存链路

目标：

1. 本阶段仍存放在 `runtime_state.json`
2. 但更新逻辑改成只 patch `runtime_state` 分片

说明：

1. 这一步先解决“别牵连整个聊天世界”。
2. 是否将来再把 runtime 拆得更细，本次不做。

3. `write_app_data()` 的调用点必须可被静态白名单枚举，禁止继续自然扩散。

## 13. 风险与注意点

## 13.1 单会话文件仍然会整文件重写

这是已接受事实。

说明：

1. 在“不改文件结构”的前提下，这是合理代价。
2. 当前主要目标是停止“重写所有分片”，而不是停止“重写当前会话文件”。

## 13.2 chat_index 与 conversation 可能出现短暂不一致

风险：

1. 若会话文件已更新但索引尚未更新，列表可能短暂滞后。

应对：

1. 明确写入顺序。
2. 第一阶段即提供具体的索引重建函数 `rebuild_index(chat_index, conversation)`，不再后置。

`rebuild_index(chat_index, conversation)` 设计要求：

1. 触发方式必须明确支持四类：
   启动时自动检查、手动 CLI/子命令触发、检测到索引写失败时登记 `index-pending` 后触发、检测到实际不一致时自动触发。
2. 默认执行语义为非阻塞：
   重建过程不默认独占业务写入，仅在显式维护模式下允许进入排他/maintenance mode。
3. 重建写入策略必须明确：
   默认采用 shadow index 或写前缓冲（write-ahead buffer）收集并发增量，重建完成后一次性切换；若实现更简单，也至少要将并发写入排队并在重建完成后回放。
4. 重建函数必须幂等：
   同一轮失败后可安全重试，不应因重复执行破坏会话或索引。
5. 锁要求必须明确：
   会话正文写入不做长时间全局阻塞，索引切换阶段才允许短暂独占。
6. 结果校验必须明确：
   重建完成后需要校验索引条数、关键字段、会话缺失/孤儿项数。

测试要求：

1. 单元测试：
   启动自动触发、手动触发、shadow index 合并逻辑、短重试与 `index-pending` 登记逻辑。
2. 集成测试：
   启动时发现不一致并修复、手动命令触发修复、重建期间有并发写入、失败后再次触发恢复。
3. 故障注入：
   会话写成功但索引写失败、重建过程中二次写入、切换 shadow index 失败。

## 13.3 兼容层可能让旧调用漏网

风险：

1. 旧接口仍在，后续开发可能继续误用。

应对：

1. 对旧接口增加注释与命名警告。
2. 新代码禁止新增整对象写入调用点。
3. CI 通过静态检查阻止新增 `write_app_data()` 调用点。
4. 兼容层调用在日志或编译期应尽量可见，避免“悄悄回退”。

## 15. 明确决策

本计划明确做以下决策：

1. 保留现有文件结构。
2. 不把 SQLite 作为本次方案。
3. `AppData` 保留，但降级为兼容聚合视图。
4. 新热路径必须按分片读写，不再按整份 `AppData` 读写。
5. 后台持久化按分片 key 合并，不再按整份快照合并。

## 16. 实施结论

本次重构不是存储介质替换，而是服务边界纠偏。

要解决的核心问题只有一个：

1. 明明已经是分文件布局，却仍然按整份应用数据做读写。

因此实现时必须坚持：

1. 不动文件结构。
2. 不新增新的整对象保存路径。
3. 优先把“整份 `AppData` 写回”改成“分片写回”。

只要这条边界改对，当前最困扰的“改一个小功能就全量写入”问题就会被直接打掉。
