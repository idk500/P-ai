# 20260423_发送前提示词准备与真相层缓存主线程爆栈分析计划

## 一、背景

当前已在本地 release 复现出一个新的聊天发送崩溃问题：

- 新会话、空历史、只发送第一条普通消息
- 前端会先出现助理草稿气泡，状态为“正在消息准备中”
- 下一瞬间应用直接闪退
- 开启 release 终端后，最终看到的是：
  - `thread 'main' (...) has overflowed its stack`

这说明：

1. 不是普通请求错误
2. 不是前端 Vue 渲染异常
3. 不是 `tokio-rt-worker` 栈溢出
4. 而是 **主线程在“发送前准备阶段”直接爆栈**

同时，已有对照事实也很关键：

- “压缩上下文”链路可成功
- 真正闪退的是“普通发送”链路
- 闪退发生在 `history_flushed`、`round_failed`、`round_completed` 之前
- `8MB Tokio worker 栈` 已生效，但对当前 `thread 'main'` 崩溃无帮助

因此，本问题不能再按“模型调用阶段 worker 栈不足”来理解，而应改为：

**发送前准备链把真相层缓存、会话快照、提示词准备结果等大对象同时挂在主线程同步持有区，导致 release 下主线程栈溢出。**

---

## 二、当前已确认事实

### 1. 崩溃发生在主线程

终端已明确打印：

- `thread 'main' has overflowed its stack`

所以当前问题不是：

- WebView2 子进程崩溃
- Tokio worker 崩溃
- 普通 `Result::Err`

而是主线程同步执行阶段直接爆栈。

### 2. 崩溃发生在“发送前准备阶段”

前端“正在消息准备中”是前端本地乐观状态，并不依赖后端首个正式事件。

当前表现为：

- 已进入发送动作
- 但尚未收到 `history_flushed`
- 也尚未收到 `round_failed / round_completed`
- 主进程就已退出

这意味着崩溃位置位于：

- `send_chat_message(...)`
- `send_chat_message_inner(...)`
- `build_prepared_prompt_for_mode(...)`

之前或其刚进入处，而不是已经进入正式流式响应后。

### 3. “压缩上下文成功”说明通用推理底座不是主因

压缩链路也会：

- 构建 prepared prompt
- 调用模型
- 产出摘要

但它不走普通发送那条前置真相准备链。

因此更合理的结论是：

- **发送独有的前置准备链有问题**
- 而不是“所有 prompt 构建和所有模型调用都天然会炸”

### 4. 发送前持久化日志缺失，说明崩在持久化/快照准备内部

当前观测到：

- `conversation_service_resolve_send_target_conversation` 的锁持有完成日志能看到
- `persist_single_conversation_runtime_fast(...)` 的尾部耗时日志看不到

这更像是：

- 已经进入发送目标会话解析
- 并且进入了“单会话快速持久化/缓存同步”
- 但在该函数内部尚未走到尾部日志时就已爆栈

---

## 三、问题定义

本轮问题不再定义为“读取大文件会不会炸”，也不定义为“模型请求本身太大”。

当前更准确的问题定义应为：

**普通聊天发送链在主线程同步准备阶段，同时持有了过厚的真相层状态与提示词层状态；这些状态经过 clone、按值传递与 release 优化内联后，形成超大调用帧 / 超大 future 状态机，最终导致主线程栈溢出。**

这里的“主线程同步准备阶段”主要包括：

1. 会话目标解析
2. 会话快写与缓存同步
3. 发送前 prompt snapshot 构建
4. prepared prompt 组装

这里的“状态重叠持有”主要包括：

1. `AppData` / `cached_app_data`
2. `Conversation`
3. `SendTargetConversationResolution`
4. `conversation_snapshot`
5. `prompt_conversation_before`
6. `PreparedPrompt`

---

## 四、核心判断

### 1. 真正的问题不是“1MB 文件读入内存”

Rust 中：

- `String`
- `Vec`
- `HashMap`

的大块数据本体本来就在堆上。

所以“文件 > 1MB”本身不能直接推出“必然 stack overflow”。

当前问题的关键不在“堆上大对象存在”，而在：

- 主线程上同时保留了太多大对象的控制状态
- 超大函数 / 超大 `async fn` 在第一个重 `await` 前已经积累大量局部变量
- 多层 clone 与按值传递把 future / 调用帧撑胖

### 2. 之前 worker 栈爆、现在 main 栈爆，本质是同一问题

之前模型请求链已有过：

- `tokio-rt-worker has overflowed its stack`

现在变成：

- `main has overflowed its stack`

这不表示问题已经换了一类，而更像是：

- 我们把一部分发送准备逻辑从后段暴露到了前段
- 或者 worker 栈被顶高后，剩余仍在主线程上的重准备逻辑被显性暴露

因此这不是两个独立问题，而是同一条发送准备链在不同线程上的不同爆点。

### 3. 当前最像“真相层缓存 + 提示词准备重叠持有”

结合日志和代码路径，当前最可疑的不是磁盘 I/O，而是：

- `resolve_send_target_conversation(...)`
- `persist_single_conversation_runtime_fast(...)`
- `sync_cached_app_data_conversation(...)`
- 后续 prompt snapshot / prepared prompt 构建

也就是：

- 先把发送目标会话及其缓存状态整理一遍
- 再把同一会话继续投影成 prompt 视图
- 两套状态在发送前主线程区间里短时间重叠

---

## 五、分析目标

本计划的目标不是“继续加大主线程栈顶住”，而是明确：

1. 发送前准备链里，哪些对象正在主线程同步持有
2. 哪些对象正在被重复 clone / 按值传递
3. 哪些逻辑应保留在真相层，哪些逻辑应延后到提示词投影层
4. 如何让“真相层缓存”和“提示词准备层”不再长时间重叠持有

最终目标是把这条链收敛成：

- 主线程只保留轻量发送上下文
- 会话真相与 prompt 投影尽量解耦
- prepared prompt 在更窄、更轻的局部作用域内构造

---

## 六、重点排查对象

### 1. 发送目标会话解析链

重点关注：

- `send_chat_message(...)`
- `conversation_service().resolve_send_target_conversation(...)`
- `persist_single_conversation_runtime_fast(...)`

要确认：

- 是否在主线程同步阶段引入了额外 clone
- 是否在缓存同步过程中产生了整份 `Conversation` / `AppData` 的再次持有

### 2. 真相层缓存同步链

重点关注：

- `state_schedule_conversation_persist(...)`
- `sync_cached_app_data_conversation(...)`
- `cached_app_data`
- `cached_conversations`
- `PendingConversationPersist`

要确认：

- 是否已经同时存在多份同一会话真相
- 是否为了异步持久化又在主线程上多保留了一层整会话副本

### 3. 提示词快照与 prepared prompt 链

重点关注：

- `build_prompt_snapshot(...)`
- `build_prepared_prompt_for_mode(...)`
- `prompt_conversation_before`
- `PreparedPrompt`

要确认：

- 会话真相是否先被 clone 成 prompt 视图
- prompt 视图是否又继续 clone 成 prepared prompt
- 是否存在“上一层大对象仍活着，下一层大对象又已构造完成”的重叠窗口

---

## 七、初步改造方向

### 0. 最高优先级硬约束

发送准备链上新增一条最高优先级约束：

- 凡是“会话”与“提示词”相关对象，默认禁止按值流动
- 默认只能以引用形式在链路中传递
- 禁止为中转、快照、局部缓存而额外创建临时持有副本
- 禁止 `clone` 会话真相、prompt snapshot、prepared prompt 这类大对象
- 禁止让这些对象作为长生命周期局部变量跨越多个阶段共存

这里的“会话 / 提示词相关对象”至少包括：

- `AppData`
- `Conversation`
- `SendTargetConversationResolution`
- `conversation_snapshot`
- `prompt_conversation_before`
- `PreparedPrompt`

唯一允许按值构造的边界应尽量收缩到：

- 最终模型请求边界
- 最终持久化写入边界

也就是说：

- 在最终消费之前，一律引用传递
- 在最终边界处，局部一次性构造并立刻消费
- 不允许把“真相层对象 -> prompt snapshot -> prepared prompt”做成多层按值串联

后续所有分析与实现，都以这条硬约束为判断标准：

- 能引用就必须引用
- 能缩短生命周期就不能长持有
- 能去掉临时变量就不能保留中间副本

### 1. 发送前主线程只保留轻量上下文

主线程发送前应尽量只保留：

- `conversation_id`
- `department_id`
- `agent_id`
- 当前用户消息
- 当前模型 ID

而不要继续长期持有：

- 整份 `AppData`
- 整份 `Conversation snapshot`
- 整份 prompt 视图

### 2. 真相层与提示词层改为“轻量投影”关系

目标不是：

- 真相层对象一路 clone 到提示词层

而应改为：

- 真相层只提供最小投影输入
- 提示词层在局部作用域内一次性消费并尽快释放

### 3. 减少发送链上的整对象 clone

优先检查并削减：

- `Conversation.clone()`
- `AppData.clone()`
- `PreparedPrompt.clone()`
- 包含大量 `messages` / `agents` 的中间快照

原则是：

- 能用引用就不用 clone
- 能用 ID 重新读取轻量视图就不传整对象
- 能缩短生命周期就不让其跨多个阶段共存

### 4. 拆小超大函数与超大 async 状态机

对超长发送入口应：

- 把“发送前会话解析”
- “发送前 prompt 投影”
- “事件入队与调度绑定”

拆成更小的 helper，降低单函数栈帧和单个 future 状态机厚度。

## 八、错误实现点清单（本轮修复范围：收到前端发送请求到调度结束）

以下问题全部位于“前端发送请求进入 Rust 命令后，到调度完成历史刷入并决定是否激活主助理”为止的链路中。

### 1. `send_chat_message(...)` / `send_user_mention_message_inner(...)` 在命令入口同步做了过厚准备

位置：

- `src-tauri/src/features/system/commands/chat_and_runtime/core_commands.rs`

问题：

- 发送入口先读配置、读整份 `AppData`、解析部门、解析会话，再构造队列事件
- 其中还会继续拿会话快照、人格列表、mention 背景
- 这导致发送命令在主线程上持有过厚的真相层状态

### 2. `resolve_send_target_conversation(...)` 把目标会话解析和“快速持久化”耦合在一起

位置：

- `src-tauri/src/features/chat/conversation_service/delegate.rs`
- `src-tauri/src/features/chat/conversation_service/persistence.rs`

问题：

- 解析目标会话时顺手执行 `persist_single_conversation_runtime_fast(...)`
- 这会在发送尚未入队、尚未调度前，就提前触发会话写入、聊天索引写入、运行态写入
- 发送链因此比压缩链多出一层“真相层缓存 + 运行态”的重叠持有

### 3. `SendTargetConversationResolution` 按值返回整份会话和整份 agents

位置：

- `src-tauri/src/features/chat/conversation_service/types.rs`

问题：

- `conversation_snapshot: Conversation`
- `agents: Vec<AgentProfile>`

这会让发送入口刚完成会话解析，就立刻再拿到一套完整会话和人格列表副本。

### 4. 发送入口为 mention 构造背景时，再次依赖按值会话快照与按值 agents

位置：

- `src-tauri/src/features/system/commands/chat_and_runtime/core_commands.rs`

问题：

- `conversation_snapshot.clone()`
- 基于 `resolved_conversation.agents` 构造 mention 背景与失败计划

这让“会话解析 -> mention 准备”之间又多出一层整对象副本。

### 5. `process_chat_event_after_ingress(...)` 在直通分支里被命令入口直接 `await`

位置：

- `src-tauri/src/features/system/commands/chat_and_runtime/core_commands.rs`
- `src-tauri/src/features/chat/scheduler.rs`

问题：

- `send_chat_message(...)` 当前直接 `await process_chat_event_after_ingress(...)`
- 若 ingress 结果为 `Direct`，会立刻在当前命令线程执行整段 `process_claimed_conversation_batch(...)`
- 而该链在真正进入模型调用前，会先完成消息准备、历史刷入、激活判定

这等于把“调度结束前”的重活继续压在命令当前线程上。

### 6. 调度层为了构造待刷入批次，额外复制了一份 agents 列表

位置：

- `src-tauri/src/features/chat/scheduler.rs`

问题：

- `state_read_app_data_cached(state)?` 后，又执行 `data.agents.clone()`
- 后续用户消息 recall 只需要借用 agents，却额外生成了一份人格列表副本

### 7. 调度提交阶段为后续持久化再次克隆整份会话

位置：

- `src-tauri/src/features/chat/conversation_service/scheduler.rs`

问题：

- `let persisted_conversation = data.conversations[conversation_idx].clone();`
- 再把这份克隆传给 `persist_after_flush(...)`

这让“历史刷入完成 -> 正式持久化”之间又多出一层整会话副本。

---

## 九、本轮修复顺序

1. 先把发送入口从“直通调度直接 await”改成“统一后台触发，命令只等结果 channel”。
2. 再拆掉 `resolve_send_target_conversation(...)` 中的前置快速持久化。
3. 把 `SendTargetConversationResolution` 收缩成只返回轻量标识，不再带整份 `Conversation` / `agents`。
4. 把 mention 背景构造改成基于当前 `AppData` 引用直接投影，不再依赖额外快照。
5. 调整调度层，去掉 `agents.clone()` 和 `persisted_conversation.clone()` 这类不必要整对象复制。

---

## 十、验证标准

当后续实现完成后，应满足：

1. 新会话、空历史、第一条普通消息在 release 下不再闪退
2. 主线程不再出现 `has overflowed its stack`
3. 不需要继续依赖 `64MB` 主线程栈才能勉强工作
4. 普通发送链路与压缩链路在 release 下都稳定
5. 发送前准备的日志阶段可清晰看到，且不再卡死于“准备中”

---

## 十一、结论

当前问题的根因判断已经足够明确：

- 不是前端气泡本身
- 不是普通 WebView2 日志
- 不是“Rust 读取 1MB 文件就会炸”
- 而是 **发送前准备阶段把真相层缓存与提示词准备层重叠持有在主线程上，导致 release 下主线程栈溢出**

因此后续工作应直接围绕：

- 真相层瘦身
- prompt 投影收窄
- clone 削减
- 主线程发送前轻量化

来展开，而不再继续把时间花在“是否是前端/是否是磁盘/是否只是 worker 栈不足”这些已经基本排除的方向上。
