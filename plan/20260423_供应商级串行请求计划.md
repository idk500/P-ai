# 20260423_供应商级串行请求计划

## 一、背景

当前聊天链路已经有两层并发控制：

- 会话级串行：同一会话同一时刻只能有一个主助理轮次
- 全局聊天并发上限：`CHAT_CONCURRENCY_LIMIT = 8`

但这两层都不是“供应商级并发控制”。

这意味着：

- 不同会话可以同时命中同一个供应商
- 不同来源的请求（聊天、远程 IM、委托、任务、归档等）也可以同时命中同一个供应商
- 对某些不稳定、限流严格、或明确不支持并发的上游接口，当前没有统一的串行门

用户诉求不是“可配置并发数”，而是更简单的：

- 在供应商设置里增加一个布尔项
- 某供应商关闭并发后，该供应商下所有模型统一串行
- 不追求“配置修改即时影响正在执行中的请求”，只要求下一次请求自然生效

---

## 二、目标

建立供应商级串行请求能力：

1. 在 `ApiConfig` 中增加 `allowConcurrentRequests` 布尔配置
2. 当该字段为 `false` 时，该供应商下所有模型请求统一串行
3. 串行门放在后端全局运行态，由后端统一维护
4. 限制点放在“实际模型请求发出前”，而不是前端，也不是仅在聊天调度器层
5. 供应商配置变更只影响后续请求，不强制影响当前已经开始的请求

非目标：

- 不做“每模型最大并发数”
- 不做前端排队器
- 不做保存配置时的锁表重建
- 不追求运行中请求的热切换

---

## 三、设计结论

### 1. 配置粒度

配置粒度定为“供应商级”，不是“模型级”：

- 配置位置：供应商设置
- 字段：`allowConcurrentRequests`
- 默认值：`true`

语义：

- `true`：允许该供应商并发请求
- `false`：该供应商下所有模型请求串行

### 2. 锁表位置

在 `AppState` 中增加全局运行态锁表，例如：

- `provider_request_gates: Arc<tokio::sync::Mutex<HashMap<String, Arc<tokio::sync::Mutex<()>>>>>`

说明：

- key 使用供应商 `providerId`
- value 为该供应商的异步串行门
- 采用惰性创建，不预先初始化整表

### 3. 触发时机

每次请求开始时，根据当前这次请求已经解析出来的供应商快照判断：

1. 解析出 `selected_api / resolved_api`
2. 读取当前这次请求的 `allowConcurrentRequests`
3. 若为 `true`，直接放行
4. 若为 `false`，按 `providerId` 获取 gate 并 `lock().await`
5. 持有 guard 覆盖整个模型请求生命周期
6. 请求完成 / 失败 / 中断后自动释放

这里的“开始时判断一次”是刻意设计，不追求运行中热更新。

### 4. 不在保存配置时重置锁表

本方案明确不在“保存供应商配置”时重置或重建锁表。

原因：

- 若某个请求已经持有旧锁，保存配置时重建锁表会产生“旧请求持有旧锁、新请求进入新锁”的分裂
- 这样反而会绕开串行约束
- 用户已经接受“配置变更影响下一次请求即可”，因此没有必要为即时生效引入复杂性

结论：

- 锁表常驻
- 惰性创建
- 每次请求按当前请求快照决定是否进入锁
- 配置变更自然作用于后续请求

### 5. 为什么不会阻塞 UI

本方案使用的是 Tokio 异步锁，而不是同步阻塞锁。

也就是说：

- `lock().await` 只会挂起当前异步任务
- 不会阻塞 UI 线程
- 不会卡住整个 Tokio worker
- 其表现是“请求排队等待 permit”，不是“线程被卡死”

## 四、推荐落点

最推荐的接入点是：

- `src-tauri/src/features/system/commands/inference_gateway.rs`

原因：

1. 这里已经是统一的模型调用策略层
2. 当前已经处理 streaming / non-stream fallback、timeout、system->user fallback
3. 这里离实际出网调用足够近，能覆盖：
   - 主聊天
   - `@提及`
   - 远程 IM
   - 任务
   - 委托
   - 归档摘要等模型调用
4. 不必侵入前端，也不必把供应商语义塞进聊天调度器

不推荐的接入点：

- 前端：拦不住后端其他入口
- `scheduler.rs`：它控制的是聊天轮次，不是供应商调用

## 五、改动范围

### 1. Rust 数据结构

需要修改：

- `src-tauri/src/features/core/domain/types_config.rs`
  - 为 `ApiConfig` 增加 `allow_concurrent_requests: bool`
  - 默认值为 `true`

- `src-tauri/src/features/core/domain/runtime_state.rs`
  - 为 `AppState` 增加供应商串行锁表字段
  - 在 `AppState::new()` 中初始化该字段

### 2. 前端类型与配置页

需要修改：

- `src/types/app.ts`
  - 为 `ApiConfigItem` 增加 `allowConcurrentRequests?: boolean`

- 配置页 API 供应商编辑界面
  - 增加一个简单开关
  - 文案建议：
    - 标签：`允许并发请求`
    - 说明：关闭后，该供应商下所有模型请求将串行执行

### 3. 推理入口

需要修改：

- `src-tauri/src/features/system/commands/inference_gateway.rs`

主要工作：

1. 新增根据 `providerId` 取或创建 gate 的 helper
2. 新增供应商串行包装 helper
3. 在统一模型调用策略入口包住真实调用过程

### 4. 测试

至少补：

- `ApiConfig` 默认值测试
- 串行门 key 生成与复用测试
- 同供应商 `allowConcurrentRequests=false` 时两个请求不能同时进入执行区
- 不同供应商之间互不影响
- `allowConcurrentRequests=true` 时不进入串行等待

## 六、实现步骤

### 第一步：补配置字段

- Rust `ApiConfig` 增加 `allow_concurrent_requests`
- TS `ApiConfigItem` 增加 `allowConcurrentRequests`
- 配置页增加开关
- 保存 / 回填链路打通

### 第二步：补运行态锁表

- 在 `AppState` 增加全局 `provider_request_gates`
- 提供统一 helper：
  - 获取 gate
  - 必要时惰性创建

### 第三步：接入推理网关

- 在 `inference_gateway` 的统一入口做供应商串行包装
- 确保持锁范围覆盖完整模型请求生命周期
- streaming / non-stream / fallback 都在同一层串行约束内

### 第四步：补日志

为排障补充聚合日志：

- 某次请求是否进入供应商串行门
- 哪个供应商正在串行
- 请求完成后释放

日志风格统一中文，例如：

- `[推理串行] 开始等待供应商串行门: provider_id=..., model=...`
- `[推理串行] 已进入供应商串行门: provider_id=..., model=...`
- `[推理串行] 完成并释放供应商串行门: provider_id=..., model=..., elapsed_ms=...`

### 第五步：回归验证

验证场景：

1. 同一供应商允许并发时，多会话可同时正常请求
2. 同一供应商关闭并发时，多会话请求会排队但不冻结 UI
3. 一个请求执行中修改供应商配置，不影响当前请求；后续请求按新配置生效
4. 不同供应商互不串行
5. 流式中断、失败、超时后能正确释放串行门

## 七、风险与应对

### 风险 1：持锁范围包得过大

如果把过多的“提示词组装 / 会话读写 / 事件发射”都包进串行门，会让排队时间明显放大。

应对：

- 串行门只包“真实供应商调用过程”
- 不把前置会话准备、历史持久化等逻辑包进去

### 风险 2：异常路径未释放

若中断、失败、超时路径没有正确释放，会导致供应商一直被锁住。

应对：

- 只用作用域 guard，不手写配对释放
- 所有返回路径依赖 Rust drop 自动释放

### 风险 3：providerId 口径不一致

若不同入口构造的供应商标识不一致，会造成同一供应商被拆成多把锁。

应对：

- 严格使用统一 `providerId`
- 不混用 `model`、`apiConfigId`、`base_url` 作为锁 key

### 风险 4：保存时重置锁表导致锁分裂

这是本方案明确规避的风险。

应对：

- 不在保存时重置锁表
- 只在请求开始时按当前配置快照决定是否进入锁

---

## 八、验收标准

满足以下条件，才算本计划完成：

1. 供应商设置页出现“允许并发请求”开关，默认开启
2. 关闭后，同一供应商下的多个请求表现为串行排队
3. 排队时应用不冻结，前后端无同步锁卡死
4. 不同供应商请求互不影响
5. streaming / non-stream / fallback / timeout / abort 都不会遗留死锁
6. 配置变更只影响后续请求，不要求影响当前请求
7. `pnpm typecheck` 与 `cargo check` 通过

---

## 九、实现后回归清单

1. 在同一供应商下连续快速发送两条消息，确认第二条等待第一条完成后才真正出网
2. 在两个不同会话里同时触发同一供应商请求，确认仍然串行
3. 在两个不同供应商上同时请求，确认仍可并发
4. 请求流式过程中点击停止，确认后续请求不会被卡住
5. 故意制造供应商报错，确认失败后下一次请求能继续进入
6. 在一个请求进行中修改“允许并发请求”开关，确认当前请求不受影响，后续请求按新设置生效
